# 下单主路径验收设计

## 背景

当前仓库已经具备以下能力：
- `user-api` / `user-rpc`：注册、登录、实名、观演人管理
- `program-api` / `program-rpc`：节目查询、预下单详情、系统分配并冻结座位
- `order-api` / `order-rpc`：创建订单、查询订单、取消订单、模拟支付、支付状态查询、超时关单
- `gateway-api`：统一 HTTP 入口和订单链路鉴权

现阶段目标不是继续扩接口，而是把“用户能够完成一次下单”收口成一条可重复执行、可验证的成功链路。

## 目标

以 `gateway-api` 作为唯一用户入口，固定一条从零开始的主路径：

1. 启动基础设施和所有必要服务
2. 注册新用户
3. 用户登录并拿到 JWT
4. 新增两个观演人
5. 查询节目预下单信息
6. 创建订单
7. 模拟支付
8. 查询订单详情和支付状态

验收时必须使用真实 HTTP 请求，不允许绕过网关直接调用内部 API 或 RPC。

## 非目标

本轮不包含以下内容：
- 前端页面或前端交互实现
- 自动化 smoke test 框架建设
- 失败分支全覆盖
- 新增业务能力或调整现有服务边界
- 非主路径的异常体验优化

## 设计方案

### 方案选择

采用“文档 + 顺序执行脚本”的验收方案：

- 一份验收文档，描述前置条件、执行顺序、每一步请求示例和预期结果
- 一个顺序执行脚本，通过 `curl` 调用 `gateway-api`
- 脚本负责保存并传递 `userId`、`token`、`ticketUserId`、`orderNumber`
- 脚本执行失败时立即停止，便于定位阻塞点

不采用“只补 README 手工步骤”，因为变量串联和排障效率较差。
不采用“直接自动化测试框架”，因为当前更需要尽快验证真实链路闭环。

### 用户入口与服务边界

验收统一走网关 `http://127.0.0.1:8081`，下游映射如下：

- 用户相关：转发到 `user-api`
- 节目相关：转发到 `program-api`
- 订单相关：转发到 `order-api`

脚本不直接访问 `8888/8889/8890`，避免验收口径与实际用户入口不一致。

### 固定验收链路

主路径固定为：

1. 注册用户
2. 登录获取 `token`
3. 添加观演人 A
4. 添加观演人 B
5. 查询用户和观演人列表，拿到两个 `ticketUserId`
6. 查询节目预下单详情，确认 `programId=10001` 的票档存在
7. 创建订单，使用两个 `ticketUserId`
8. 查询订单详情，确认订单和系统分配座位已生成
9. 发起模拟支付
10. 查询支付状态
11. 再次查询订单详情，确认订单已支付

### 数据约束

验收依赖当前仓库已有 SQL 和种子数据：

- 用户域表结构
- 节目域表结构与 `dev_seed.sql`
- 订单域表结构
- 支付域表结构

节目验收默认使用种子节目 `programId=10001`。票档 ID 不在文档中硬编码假设，而是优先通过节目预下单详情或票档查询结果确认；若当前种子数据稳定，也可在脚本中使用仓库已验证过的 `ticketCategoryId=40001`。

### 鉴权规则

订单链路必须同时携带：

- `Authorization: Bearer <token>`
- `X-Channel-Code: 0001`

用户链路和节目链路默认不要求 JWT，但登录接口需要在请求体中提供渠道码 `code=0001`。

### 执行产物

计划新增两类产物：

1. 验收文档
   - 描述前置环境
   - 给出顺序步骤
   - 给出每一步的成功判定

2. 执行脚本
   - 自动发起 HTTP 请求
   - 自动解析关键字段
   - 在终端打印阶段结果
   - 失败即停止

## 成功标准

验收通过需满足：

- 注册返回 `success=true`
- 登录返回非空 `userId` 和 `token`
- 新增两个观演人成功
- 查询用户与观演人列表时能拿到新增观演人的 ID
- 能查询到节目预下单详情和有效票档
- 创建订单成功并返回 `orderNumber`
- 订单详情中包含观演人、票档、座位和未支付状态
- 模拟支付成功
- 支付查询返回已支付状态
- 最终订单详情显示已支付

## 风险与排障重点

优先关注以下风险：

1. 基础设施未完全启动
   - `etcd`
   - `mysql`
   - `redis`

2. SQL 未完整导入
   - 节目种子数据缺失
   - 座位/票档数据缺失
   - 订单表或支付表未初始化

3. 请求链路变量依赖
   - 观演人 ID 不能手写，必须动态获取
   - `orderNumber` 必须由创建订单响应回填

4. 鉴权请求头遗漏
   - 订单接口缺 `Authorization`
   - 订单接口缺 `X-Channel-Code`

## 测试与执行策略

本轮先采用手工可执行、脚本可复现的方式验收：

- 先确认基础设施和服务状态
- 再按主路径逐步执行
- 任一步失败时，当场定位并修复阻塞点
- 闭环跑通后，再决定是否补自动化 smoke test

## 实施后的下一步

若本轮主路径验收稳定，下一阶段可按优先级继续：

1. 补失败分支验收
2. 补脚本化一键启动与一键验收
3. 补自动化 smoke test
4. 再评估是否需要最小演示前端
